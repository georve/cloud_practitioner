AWS Service Catalog permite a las organizaciones crear y administrar catálogos de servicios de TI aprobados para su uso en AWS. Estos servicios de TI pueden incluir todo, desde imágenes de máquinas virtuales, servidores, software y bases de datos hasta arquitecturas de aplicaciones completas de varios niveles. AWS Service Catalog le permite administrar de forma centralizada los servicios de TI implementados comúnmente y le ayuda a lograr una gobernanza coherente y cumplir con sus requisitos de conformidad, al tiempo que permite a los usuarios implementar rápidamente solo los servicios de TI aprobados que necesitan.

AWS Organizations proporciona administración y gobierno centrales en varias cuentas de AWS. Las políticas de control de servicios de AWS (o políticas de AWS Organizations) son un tipo de política de organización que puede utilizar para administrar los permisos de todas las cuentas de su organización. Los SCP ofrecen un control central sobre el máximo de permisos disponibles para todas las cuentas de miembros de su organización. Los SCP le ayudan a garantizar que las cuentas de los miembros se mantengan dentro de las directrices de control de acceso de su organización. En los SCP, puede restringir a qué servicios, recursos y acciones de API individuales de AWS pueden acceder los usuarios y roles de cada cuenta de miembro. Cuando AWS Organizations bloquea el acceso a un servicio, recurso o acción de API para una cuenta de miembro, un usuario o rol de esa cuenta no puede acceder a ella. Este bloqueo permanece en vigor incluso si un administrador de una cuenta de miembro concede explícitamente dichos permisos en una política de IAM.
Información adicional:
¿Cuál es la diferencia entre una política de control de servicios (SCP) de AWS Organizations y una política de IAM?
Una política de IAM proporciona un control granular sobre lo que pueden hacer los usuarios y roles de cuentas individuales. AWS Organizations amplía ese control al nivel de cuenta al brindarle control sobre lo que pueden hacer los usuarios y roles de una cuenta o un grupo de cuentas. Los permisos resultantes son la intersección lógica de lo permitido por AWS Organizations en el nivel de cuenta y los permisos que IAM concede explícitamente en el nivel de usuario o rol dentro de esa cuenta. En otras palabras, el usuario solo puede acceder a lo permitido tanto por las políticas de AWS Organizations como por las políticas de IAM. Si alguno de los dos bloquea una operación, el usuario no puede acceder a esa operación. Por ejemplo, si un SCP aplicado a una cuenta indica que las únicas acciones permitidas son las acciones de Amazon EC2 y los permisos de una entidad de seguridad (usuario o rol de IAM) en la misma cuenta de AWS permiten tanto acciones de EC2 como acciones de Amazon S3, la entidad de seguridad solo puede obtener acceso a las acciones de EC2.

Las instancias dedicadas son instancias Amazon EC2 que se ejecutan en una nube privada virtual (VPC) en hardware dedicado a un único cliente. Las instancias dedicadas que pertenecen a diferentes cuentas de AWS se aíslan físicamente en el nivel de hardware. Además, las instancias dedicadas que pertenecen a cuentas de AWS que están vinculadas a una sola cuenta de pagador también están físicamente aisladas a nivel de hardware. Sin embargo, las instancias dedicadas pueden compartir hardware con otras instancias de la misma cuenta de AWS que no sean instancias dedicadas.

Amazon Aurora es un motor de base de datos relacional totalmente administrado, compatible con MySQL y PostgreSQL. Combina la velocidad y confiabilidad de las bases de datos comerciales de alta gama con la simplicidad y rentabilidad de las bases de datos de código abierto. Ofrece hasta cinco veces el rendimiento de MySQL y hasta tres veces el rendimiento de PostgreSQL sin requerir cambios en la mayoría de sus aplicaciones existentes.

Un caso de negocio es el primer paso en su viaje de migración. Crear casos de negocio por su cuenta puede llevar mucho tiempo y no siempre identifica las opciones de implementación y compra menos costosas. AWS Migration Evaluator es un servicio de evaluación de migración que le ayuda a crear un caso de negocio direccional para la planificación y migración de la nube de AWS.
Migration Evaluator analiza su huella informática local, incluida la configuración del servidor, la utilización, los costos anuales de operación, la elegibilidad para traer su propia licencia y cientos de otros parámetros. Después de la recopilación de datos, recibirá rápidamente una evaluación que incluye una estimación de costos proyectada y ahorros de la ejecución de sus cargas de trabajo locales en la nube de AWS. Después de recibir su evaluación inicial, su organización puede trabajar con el equipo de Migration Evaluator para crear un caso de negocio direccional que mejor se adapte a los requisitos de su organización.

AWS Snowmobile es un servicio de transferencia de datos a escala de exabytes que se utiliza para mover cantidades extremadamente grandes de datos a AWS. Puede transferir hasta 100 Petabytes (PB) por moto de nieve, un contenedor de envío robusto de 45 pies de largo, tirado por un camión semirremolque. Snowmobile facilita el traslado de volúmenes masivos de datos a la nube, incluidas bibliotecas de video, repositorios de imágenes o incluso una migración completa del centro de datos. A escala de exabytes, la transferencia de datos con Snowmobile es más segura, rápida y rentable.

Proporciona una capacidad de cómputo (y\o almacenamiento) rentable y redimensionable al tiempo que automatiza las tareas de administración que consumen mucho tiempo, como el aprovisionamiento de hardware, el mantenimiento del sistema operativo, la configuración de la base de datos, la aplicación de parches y las copias de seguridad. Le libera para centrarse en sus aplicaciones para que pueda darles el rendimiento rápido, la alta disponibilidad, la seguridad y la compatibilidad que necesitan.

Amazon Web Services (AWS) permite a los clientes asignar metadatos a sus recursos de AWS en forma de etiquetas. Cada etiqueta es una etiqueta simple que consta de una clave definida por el cliente y un valor opcional que puede facilitar la administración, la búsqueda y el filtrado de recursos. Aunque no hay tipos inherentes de etiquetas, permiten a los clientes categorizar los recursos por propósito, propietario, entorno u otros criterios. Una estrategia de etiquetado efectiva le brindará una mejor visibilidad y monitoreo, lo ayudará a crear modelos precisos de devolución de cargo / devolución de información y obtendrá información más detallada y precisa sobre el uso y el gasto por parte de las aplicaciones y los equipos.

Para habilitar las conexiones HTTPS a su sitio web o aplicación en AWS, necesita un certificado de servidor SSL/TLS. Puede utilizar un certificado de servidor proporcionado por AWS Certificate Manager (ACM) o uno que haya obtenido de un proveedor externo. Puede usar ACM o IAM para almacenar e implementar certificados de servidor. Utilice IAM como administrador de certificados solo cuando deba admitir conexiones HTTPS en una región que no sea compatible con ACM. IAM admite la implementación de certificados de servidor en todas las regiones, pero debe obtener el certificado de un proveedor externo para su uso con AWS. Amazon Route 53 se utiliza para registrar nombres de dominio o utilizar su propio nombre de dominio para enrutar a los usuarios finales a las aplicaciones de Internet. Route 53 no es responsable de crear certificaciones SSL.
